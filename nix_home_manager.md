### 1. The `bootstrap.sh` Script

This script is the orchestrator. You would place it in the root of your dotfiles repository on GitHub. It's designed to be idempotent, meaning you can run it again without breaking things.

`write_file`
**path**: `bootstrap.sh`
**content**:
```bash
#!/usr/bin/env bash
#
# Bootstrap script for setting up a new system with Nix, Home Manager, and dotfiles.
# This script is idempotent and can be safely run multiple times.

set -euo pipefail

# --- Configuration ---
# Replace with your GitHub username and repository name.
GITHUB_USER="your-username"
REPO_NAME="dotfiles"
DOTFILES_DIR="$HOME/.dotfiles"

# --- Helper Functions ---
info() {
    echo -e "\033[34m[INFO]\033[0m $1"
}

success() {
    echo -e "\033[32m[SUCCESS]\033[0m $1"
}

warn() {
    echo -e "\033[33m[WARN]\033[0m $1"
}

# --- Main Logic ---

# 1. Install Nix
if ! command -v nix &> /dev/null; then
    info "Nix not found. Installing Nix..."
    # The multi-user installer is recommended for most systems.
    sh <(curl -L https://nixos.org/nix/install) --daemon
    
    # Source the Nix environment script to make `nix` command available in this session.
    # This is crucial for the rest of the script to work.
    # shellcheck source=/dev/null
    . /nix/var/nix/profiles/default/etc/profile.d/nix-daemon.sh
    success "Nix installed successfully."
else
    info "Nix is already installed. Skipping installation."
fi

# 2. Enable Nix Flakes
# The configuration path can vary, but this is a common user-level setup.
NIX_CONFIG_DIR="$HOME/.config/nix"
NIX_CONFIG_FILE="$NIX_CONFIG_DIR/nix.conf"
mkdir -p "$NIX_CONFIG_DIR"
if ! grep -q "experimental-features" "$NIX_CONFIG_FILE" 2>/dev/null; then
    info "Enabling Nix Flakes..."
    echo "experimental-features = nix-command flakes" >> "$NIX_CONFIG_FILE"
    success "Nix Flakes enabled."
else
    info "Nix Flakes are already enabled."
fi

# 3. Clone Dotfiles Repository
info "Checking for dotfiles repository at $DOTFILES_DIR..."
if [ -d "$DOTFILES_DIR" ]; then
    info "Dotfiles repository already exists. Pulling latest changes..."
    git -C "$DOTFILES_DIR" pull
else
    info "Cloning dotfiles repository..."
    git clone "https://github.com/$GITHUB_USER/$REPO_NAME.git" "$DOTFILES_DIR"
fi
success "Dotfiles repository is up to date."

# 4. Run Home Manager
info "Applying Home Manager configuration..."
# This command tells Nix to run the `switch` command from the `home-manager` flake,
# and to use the configuration defined in your dotfiles flake (`.#$USER@$HOSTNAME`).
# The target host must be defined in your flake.nix.
nix run home-manager/master -- switch --flake "$DOTFILES_DIR#$USER@$(hostname)"
success "Home Manager configuration applied."

# 5. Symlink remaining dotfiles with Stow
# This step is for files that are not managed by Home Manager.
info "Symlinking dotfiles with GNU Stow..."
cd "$DOTFILES_DIR"
# The `stow` command will symlink the contents of each specified directory
# into the parent directory of the target (-t $HOME).
# For example, it will link .dotfiles/stow-packages/nvim/.config/nvim to ~/.config/nvim
stow -vSt "$HOME" stow-packages/*
success "Stow symlinking complete."

echo -e "\n\033[32m🚀 Bootstrap complete! Please restart your shell to see all changes.\033[0m"

```

### 2. The `flake.nix` File

This is the heart of your declarative setup. It defines which packages to install and how to configure them. You would place this file in the root of your `dotfiles` repository.

`write_file`
**path**: `flake.nix`
**content**:
```nix
{
  description = "My Home Manager configuration";

  inputs = {
    # Pin Nix packages to a specific commit for reproducibility.
    nixpkgs.url = "github:NixOS/nixpkgs/nixos-unstable";
    
    # Home Manager, following the same Nixpkgs version.
    home-manager = {
      url = "github:nix-community/home-manager";
      inputs.nixpkgs.follows = "nixpkgs";
    };
  };

  outputs = { self, nixpkgs, home-manager, ... }@inputs:
    let
      # Define the system architecture.
      system = "x86_64-linux";
      
      # Create a pkgs set for the specified system.
      pkgs = nixpkgs.legacyPackages.${system};
      
      # Define your username and hostname.
      # You can add multiple configurations for different machines.
      username = "your-username";
      hostname = "your-hostname"; # Get this by running `hostname` on the target machine.

    in {
      # The `homeConfigurations` output is the entry point for Home Manager.
      homeConfigurations."${username}@${hostname}" = home-manager.lib.homeManagerConfiguration {
        inherit pkgs;

        # Modules are the building blocks of your configuration.
        modules = [
          {
            # --- User and System Settings ---
            home.username = username;
            home.homeDirectory = "/home/${username}";
            home.stateVersion = "24.05"; # Set to the current Home Manager version.

            # --- Packages to Install ---
            # This is where you list all the command-line tools you want.
            home.packages = with pkgs; [
              # Essentials
              git
              stow # For managing other dotfiles
              neovim
              ripgrep
              fd
              zsh
              
              # Fun
              cowsay
            ];

            # --- Program Configurations ---
            # Home Manager has modules for many common programs.
            # This is often better than stowing config files.

            programs.git = {
              enable = true;
              userName = "Your Name";
              userEmail = "your.email@example.com";
            };

            programs.zsh = {
              enable = true;
              # You can manage the .zshrc file directly.
              # Here, we source it from a file within the repo.
              initExtra = builtins.readFile ./home-config/zshrc;
            };
            
            # Let Home Manager manage itself
            programs.home-manager.enable = true;
          }
        ];
      };
    };
}
```

### 3. Proposed Dotfiles Directory Structure

This structure organizes your files cleanly, separating what Nix manages from what `stow` manages.

```
/home/user/dotfiles/
├── bootstrap.sh          # The script to kick everything off
├── flake.nix             # The master Nix configuration
├── flake.lock            # Generated by Nix to pin dependencies
├── README.md
|
├── home-config/          # Files managed directly by Home Manager
│   └── zshrc             # Example: your .zshrc content
|
└── stow-packages/        # Directories for stow to manage
    ├── nvim/             # Stow package for Neovim
    │   └── .config/
    │       └── nvim/
    │           └── init.lua  # Stow will link this to ~/.config/nvim/init.lua
    └── tmux/             # Stow package for Tmux
        └── .tmux.conf    # Stow will link this to ~/.tmux.conf
```

### 4. How to Execute It

With the `bootstrap.sh` and `flake.nix` files pushed to your `dotfiles` repository on GitHub (e.g., at `github.com/your-username/dotfiles`), you would run the following command on your new, freshly installed system:

```sh
curl -L https://github.com/your-username/dotfiles/raw/main/bootstrap.sh | bash
```

**How it works:**
1.  `curl` downloads the `bootstrap.sh` script from your repository.
2.  The pipe `|` sends the script's content directly to `bash`.
3.  `bash` executes the script, which then handles the entire installation and configuration process you defined.

This setup provides a powerful, reproducible, and automated way to bring any new machine to your preferred state with a single command.
